最新需要复习一下网络相关的知识，顺手做一下笔记。
找工作的时候，经常遇到面试官问：“在浏览器地址栏输入网址，敲一下回车，发生了什么？”

模型

OSI 7 层模型和 TCP/IP 4 层模型。
所谓 OSI，意为 open system interconnection 开放式系统互联参考模型。

工具

笔者使用 netwox 和 wireshark 进行学习。

CDN

如果访问的是一个静态资源，那么可能会访问 CDN。

DNS

DNS（Domain Name System，域名系统），当在浏览器输入地址，比如https://www.baidu.com，最终访问的是 ip 地址。将域名转换为 ip 的过程称之为 dns 解析。

  1. 查找浏览器缓存
    浏览器本身会缓存 dns 记录，缓存时间大概在 2 ～ 30 分钟左右。找到对应的 ip 就返回，找不到执行下一步。
  2. 查找系统缓存
    如果浏览器缓存中没有对应的 dns 记录，就会访问系统的 hosts 文件。找到对应的 ip 就返回，找不到执行下一步。
  3. 查找 dns 服务器
    向本地 dns 服务器发起请求，如果本地 dns 服务器没有缓存对应的 ip，则依次向跟域名服务器、顶级域名服务器，发起请求，直至查询到 ip 地址，返回给浏览器。

HTTP DNS

DHCP

动态主机配置协议，ip 地址可以手动设置，当然一直手动设置比较麻烦，于是就有了 dhcp 协议。
当新机器加入一个网络的时候，使用 0.0.0.0 作为 ip 地址，发送一个广播包。目的 ip 地址为 255.255.255.255。
dhcp server 通过 mac 地址判断，这是个新机器，就会给它分配一个 ip 地址。
有时候 dhcp server 有多个，新机器可能收到多个 ip。新机器接收到 ip 之后，会发送一个广播包。告诉所有 dhcp server，自己已经有 ip 了。

MAC

MAC 全称 Medium Access Control，意为媒体访问控制。
通常说的 MAC 指的是 MAC 地址，类似于设备的身份证号码。比如 81:0f:1f:df:d6:89，用 16 进制表示，6 个 byte。共 48bit，前 24bit 表示厂商，后 24bit 由厂商自己分配。一般一个 MAC 地址对应着唯一的设备，可以通过https://mac.51240.com/查询设备的生产厂商。
Q：竟然 mac 地址唯一，为什么还需要 ip 地址。
A：mac 地址通讯范围有限，只能在一个子网里通信。比如我知道一个人的身份证号，但是我并不能通过身份证号知道他住在哪里。

查看本机网络信息（netwox 169）

通过在命令行执行netwox 169，可以看到本机的信息。Lo 表示回环接口，它是虚拟网络适配器，Eth 是以太网网络适配器，多个的话在后面添加数字编号。

查看网段网络信息（netwox 5）

通过netwox 5 -i xxx.xxx.xxx.xxx，可以查看对应 ip 地址的机器 mac 地址。ip 参数可以是固定的 ip，也可以是某个网段。还可以添加-u参数查看进度，如果存在会显示 mac 地址，如果不存在，会显示 unresolved。

根据 MAC 地址查信息（netwox 4）

通过netwox 4 -e xx:xx:xx:xx:xx:xx，可以根据 mac 地址查到对应机器信息。

Ethernet

以太网传输的数据包称为“以太帧”。
以太帧起始部分，由 7 个 byte 的“前同步码”和 1 个 byte 的“帧开始定界符”组成。后面紧跟的是“以太网 MAC 帧”，其中 6 个 byte“目的 Mac 地址”，6 个 byte“源 Mac 地址”，2 个 byte 的“类型”（比如类型 0800：IP 数据报，交给 ip 协议。类型 0806：ARP 请求，应答）。再后面 46 ～ 1500byte 是“ip 数据报”，不足 46 字节时，会填充到最小长度。最大值也叫最大传输单元（MTU）,ifconfig 可以查到这个值，通常为 1500。最后 4byte 是“帧检测”，主要用于校验传输过程中，数据是否丢失或者改变。

查看以太帧（netwox 32）

通过 netwox 32 查看以太帧。60:F8:1D:C8:58:B7 为源 MAC 地址，00:08:09:0A:0B:0C 为目标 MAC 地址。0x0000 是以太网类型。

这个命令通过添加参数，可以手动构造以太帧。-a为源 MAC 地址，-b为目标 MAC 地址。

通过 Wireshark 抓包，可以看到目标地址 mac 确实是 01:02:03:04:05:06。

构造以太帧攻击（netowx 75）

交换机内部有张表，存储了 Mac 地址，方便转发数据。当然这张表大小是有限的，如果存满了，将不会再处理新来的请求。netwox 75 可以实现以太洪水攻击，伪造大量的以太数据包，填满交换机的存储表，使其失去正确的转发功能。

通过 Wireshark 可以发现，有大量的伪造数据。

显示网络信息（netwox 1,2）

netwox 1 显示网路信息。
netwox 2 显示网路调试信息。

IP

ip 地址类似现实世界中的门牌号码。常见的有 IPv4 协议，IPv6 协议。
IPv4 协议比如本地 ip：127.0.0.1。通过点号分割为 4 个部分，每个部分 8 个 bit，共 32bit。
IPv6 协议比如 fe80::f816:3eff:fec7:7975/64，一共 128bit。
由于 IPv6 尚未普及，为了缓解 IPv4 地址数量不够的问题，引入了 CIDR（无类型域间选路）。比如 10.100.122.2/24，ip 地址后面有个/24，表示 32bit 中，前 24bit 代表网络号，后 8bit 代表主机号。

ip 协议包结构


ip 协议包，包含两部分，“首部”和“数据部分”。“首部”最小长度为 20 字节。

  • 版本（version）：占 4 位，表示 ip 协议的版本。比如“0100”表示 IPv4。
  • 首部长度（header length）：占 4 位，可表示最大十进制数值是 15（即 1111），这个时候“首部”长度就是 60 字节。常用的数值是 20 字节，即 0101。
  • 区分服务（tos）：也称之为“服务类型”，占 8 位。区分服务时，此字段才有用。包括 3 bit 的优先权字段（取值可以从 000-111 所有值），4 bit 的 TOS 子字段和 1 bit 未用位但必须置 0。
  • 总长度（total length）：”首部”和“数据部分”长度之和，单位是字节。占 16 位，所以整个数据报文最大值 2^16-1=65535 字节。
  • 标识（identification）：用来标识数据报，占 16 位。IP 协议在存储器中维护了一个计数器，每产生一个数据报，计数器加 1，并将此值赋值给“标识”字段。当数据报的长度超过网络的 MTU，而必须分片时，这个标识字段的值就被赋值到所有的数据报文字段中。具有相同标识字段值的分片报文，会重组成原来的数据报。
  • 标志（flags）：占 3 位。第一位未使用。第二位称为“don’t fragment”，0 表示允许分片，1 表示不允许分片。第三位称为“more fragments”，0 表示没有更多的分片需要发送，或者表示没有分片。
  • 片偏移（fragment offset）：占 13 位。报文被分片后，该字段标志该分片在原报文中的相对位置。片偏移以 8 个字节为片偏移单位。所以，除了最后一个分片，其他分片都是 8 字节（64 位）的整数倍。
  • 生存时间（time to live）：占 8 位，表示数据报在网络中的寿命。每经过一个路由器，路由器会将 TTL 数值减 1，直至位 0 时，丢弃数据，不再转发。最大值 255，如果设置为 1，表示该数据只能在本局域网中传送。
  • 协议（protocol）：表示数据报文携带数据使用的协议，方便目的主机 ip 层按照协议解析数据，占 8 位。比如 TCP 协议号是 6，UDP 的协议号是 17，ICMP 的协议号是 1。
  • 首部校验和（header checksum）：用来校验“首部”数据，占 16 位。数据报经过路由器时，比如 TTL 可能会发生变化，所以这个值也会跟着变化。
  • 源地址（source）：表示数据报的源 ip 地址，占 32 位。
  • 目的地址（destination）：表示数据报的目的地址，占 32 位。
  • 可选字段：该字段用于一些可选的设置。
  • 填充：由于可选字段的长度不固定，所以需要填充 0，使长度是 32 位的倍数。
  • 数据部分：用于传输的数据。

TCP 报文格

  • 源端口（Source Port）：源计算机上的应用程序端口号，占 16 位。
  • 目的端口（Destination Port）：目的计算机的应用程序端口号，占 16 位。
  • 序列号（Sequence Number）：占 32 位。用于表示本报文段发送数据的第一个字节编号。
  • 确认号（Acknowledgment Number）：占 32 位。用于表示接收方期望发送方下一个报文段的第一个字节数据的编号。
  • 首部长度（Header Length）：占 4 位，表示“首部”的长度。
  • 保留（Reserved）：占 4 位，保留位置，必须全部是 0。
  • 标识位
    CWR（Congestion Window Reduce）：拥塞窗口减少标志，用来表明它接收到了设置 ECE 标志的 TCP 包。并且，发送方收到消息之后，通过减小发送窗口的大小来降低发送速率。
    ECE（ECN Echo）：用来在 TCP 三次握手时表明一个 TCP 端是具备 ECN 功能的。在数据传输过程中，它也用来表明接收到的 TCP 包的 IP 头部的 ECN 被设置位 11，即网络线路拥堵。
    URG（Urgent）：表示本报文段中发送的数据是否包含紧急数据。URG=1 时表示有紧急数据，且只有 URG=1 时，后面的“紧急指针”字段才有效。
    ACK：ACK=1 时，表示有效。TCP 规定，连接建立后，ACK 必须为 1。
    PSH（Push）：如果值为 1 的话，表示把数据立即发送给上层，而不是缓存起来。
    RST：如果值为 1 的话，表示 TCP 连接出现了严重的错误，必须重新建立连接。
    SYN：在建立连接时使用，用来同步序号。当 SYN=1，ACK=0 时，表示这是一个请求建立连接的报文段。当 SYN=1，ACK=1 时，表示对方同意建立连接。
    FIN：如果值为 1 的话，表示数据发送完毕，可以释放连接。
  • 窗口大小（Window Size）：占 16 位。它表示从 Ack Number 开始还可以接收多少字节的数据量，也表示当前接收段的接收窗口还有多少剩余空间。该字段可以用于 TCP 的流量控制。
  • TCP 校验和字段（TCP checksum）：占 16 位。用于校验传输的数据是否有损坏。
  • 紧急指针（Urgent Pointer）：占 16 位。仅当“URG”设置位 1 时才有意义。表示数据段中紧急数据的字节数。
  • 选项（Option）：长度不定，但必须是 32 位的整数倍。

三次握手

三次握手的过程。

通过 wireshark 抓包，详细地看一下 3 次握手过程。
第 1 次握手，SYN=1，seq=0。

第 2 次握手，SYN=1，ACK=1，ack=1。

第 3 次握手，seq=1，ack=1，ACK=1。

Q：为何要 3 次握手？
A：为了防止已经失效的请求连接报文段突然又传到服务端，因而产生错误。

构造 3 次握手包（netwox 42）

通过 netwox 42 命令可以构造 3 次握手包，-n 2表示构造第 1 次握手包，-n 3表示构造第 2 次握手包，-n 4表示构造第 3 次握手包。

数据传输

数据分片

发送的数据包，有最大字节数限制，称之为 MMS（maximum segment size）。当发送的数据超过这个大小的时候，需要分片。第 1 次和第 2 次的握手包，会包含这个 MMS，通知服务器和客户端各自的 MMS。

滑动窗口机制

如果每发送一个包，都需要接收端应答一次的话，比较浪费时间。TCP 引入了“窗口”概念，比如设置了窗口大小为 3，发送端可以直接发送 3 个数据包，不用等待。待 3 个数据包发送完，等待接收端应答这 3 个数据包，接着发送接下来的 3 个数据包。如果发送端没有收到某个响应包，则会调整窗口，继续发送没收到对应响应包的数据包。

数据重发

如果发送了 3 个数据包，前 2 个数据包没有收到应答包，最后 1 个应答包收到了。则默认前 2 个数据包接收端收到了，不会重发。
如果接收端确实没有收到应答包，会重复 3 次确认应答包。这时发送端会重新发送数据。

TCP 流控制

为了合理利用资源，当窗口过小或者过大的时候，会在数据发送和应答时，调整窗口大小。

四次挥手

断开连接，还会涉及到 4 次挥手。

通过 wireshark 抓包，详细地看一下 4 次挥手过程。
第 1 次挥手，FIN=1，ACK=1，seq=x。

第 2 次挥手，ACK=1，seq=y，ack=x+1。

第 3 次挥手，FIN=1，ACK=1，seq=z，ack=x+1。

第 4 次挥手，seq=x+1，ack=z+1，ACK=1。

Q：为什么建立连接需要 3 次握手，断开连接需要 4 次。
A：server 在 LESTEN 状态下，收到建立连接请求的 SYN 报文后，把 ACK 和 SYN 放在同一个报文发送给 client。而关闭连接时，当收到 client 的 FIN 时，仅表示 client 不再发送数据，但还能接收数据。server 也可能没有把所有数据发送完毕，所以先 ACK，再发送 FIN。
