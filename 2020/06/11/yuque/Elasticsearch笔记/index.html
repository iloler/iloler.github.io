背景

使用的场景

因业务需要加入关键字搜索功能，所以引入了Elasticsearch（简称es）。
源数据存储在mysql中，新增数据时候，通过接口同步将数据放入es。抽象一下，大致有以下几张表。
question_content（题目内容表）

字段名 数据类型 备注
id bigint 主键
title text 题干
option_a varchar 选项A
option_b varchar 选项B
option_c varchar 选项C
option_d varchar 选项D
answer varchar 答案
analysis text 解析
create_time timestamp 创建时间

question_tag（题目标签表）

字段名 数据类型 备注
id bigint 主键
question_id bigint 题目主键
grade_id int 年级id
subject_id int 学科id
score double 分数

question_knowledge（题目和知识点对应关系表）

字段名 数据类型 备注
id bigint 主键
question_id bigint 题目主键
knowledge_id bigint 知识点主键

knowledge（知识点表）

字段名 数据类型 备注
id bigint 主键
name varchar 知识点名称

将这几张表的数据整合一下，合并成一个索引，放入es中。按照关键字查询的时候，使用match就完事了。

业务不满意

平时局限于能用就行，也没时间好好研究一下。最近因为业务对于搜索结果不满意，遂准备着手优化一下。

分析器

直接使用默认的分析器analyzer，或许不能满足需求，需要定制一个。本人所在的组代号TK，那么自定义的分词器就叫tk_analyzer吧。
默认的analyzer包含了3块，分别是character filters, tokenizers, token filters。

character filters

预处理

  • html_strip这个filter可以去掉html，比如文本是<p>I'm so <b>happy</b>!</p>，过滤完变成了\nI'm so happy!\n。公司的业务使用了富文本编辑内容，所以会有部分的html标签，以及一些自定义的标签，比如这种。貌似有个escaped_tags参数，表示不需要过滤的html标签。
  • mapping character filter可以将字符做一个映射，比如把:)映射成happy。

  • pattern replace正则匹配替换。

    tokenizers

    分词
    显然笔者不会去使用standard，既然安装了ik插件，当然使用ik分词了。之前用的ik_smart，感觉不太好。这次选择ik_max_word试试。还有个参数max_token_length可以设置一下，默认255。

    token filters

    分词处理

  • Lowercase token filter，顾名思义，将原文本规范为小写字母。

  • Stop token filter，停止词，表示忽略这些词，常见的英语里的this，that，the，中文里的“的”，“地”，“得”。

映射（mapping）

通过以下命令，设置一下mapping。对于文本类型的数据格式，最好严格区分text和keyword类型，默认会使用string类型，同时使用（text,keyword）占了额外的空间，有点浪费空间。需要分词的字段analyzer设置为ik_smart和ik_max_word，ik_smart分词粒度较粗，ik_max_word分词粒度较细。根据实际使用场景选择。
注：测试问我，为什么输入一个数字“8”，查不到名为“88888888”的试卷。因为试卷名称name字段analyzer设置的ik_smart，考虑换成ik_max_word。搜索的时候，结合match_phrase和slop调整顺序和精度。同时，对text再设置一下standard，切割为单个汉字，提高业务查询的满意度。

copy_to

实际需求中，题目有关联一个主知识点，N个副知识点。建立mapping的时候，可以在mainKnowledge和knowledge字段，加上copy_to属性。以后查询知识点，只需要根据“knowledgeIdList”查询即可。注：“knowledgeIdList”这个字段不在返回的“_source”里。

动态映射（dynamic mapping）

如果没有手动设置mapping的字段类型的话，es会根据json数据的格式，自动设置mapping的字段类型。我觉得设置mapping的时候，最好把dynamic设置为strict，新字段写入直接报错，保险一点。

搜索

衡量相关性

1.查准率（precision），尽可能返回较少的无关文档。
2.查全率（recall），尽量返回较多的相关文档。
3.排序（ranking），按照相关度进行排序。

URI Search

在URL参数中添加查询参数，比如查询gradeId=1的数据。这种也就查询参数少，可以用一下，多了要疯。

Request Body Search

GET或者POST都可以。

match

目前业务中，使用的是这个match，配合minimum_should_match。然后分词还是用的ik_smart，效果贼差，天天被业务骂。可以加上“operator”参数，设置为“AND”，这样就查询“中国”两个字连在一起的文档。

match_phrase

网传最常用的查询用词，配合slop使用。slop的值大概就是允许查询的词中间间隔的距离。

term

不分词，严格匹配，完全一样才行。

terms

匹配多个数值。

filter

配合constant_score不打分，很快返回数据，有效利用缓存。

range

范围查询，大于等于，小于之类的。比如查询1年级到9年级的数据，不包括9年级。

explain

可以看到算分的过程。

should是or的关系，must和must_not是and的关系。

多值

一个字段，在文档中，有些数据是单值，有些是多值，这个时候要精确匹配比较麻烦。
比如一个题目可以有一个知识点，也可以有多个知识点。那么有这样一个需求，“精确查询有理数这个知识点下的题目，如果题目包含了其他知识点就不要。”。对于这种需求，我们可以增加一个字段，knowledgeCount记录知识点个数。可以达到精确匹配的目的。

dis_max

单字符串多字段查询的时候，会用should，但是should打分计算方式返回的结果，可能不符合预期。这个时候就需要引入dis max query了。有个tie_breaker参数可以用一下，这个参数默认为0，最大值为1。使用它，主要是为了增加其他字段起的作用。

multi_match

另一个用来查单字符串多字段的用法。type有best_fields，most_fields，cross_fields（最佳字段、多数字段、跨字段）。

search_template

新建一个search_template。

通过search_template执行查询语句。

function score query

有时候es打分生成的排序，不一定满足我们的需求。比如我希望随机符合查询条件的数据，就可以用random score。seed可以设置为userId，不同的人可以查到不同的随机数据。

suggester

特殊类型的搜索。比如输入了一个错误的词汇“helo”。suggest_mode有missing（如果索引中已经存在，就不提供建议。），popular（推荐出现频率更加高的词），always（无论是否存在，都提供建议）。
除了term-suggestion，还有phrase-suggestion。

es返回的suggest中，告知我们“hello”，还有几个可选项。

completion suggester

自动补全功能，这玩意对性能要求挺高，es将analyze的数据编码成FST，和索引一起存放，并且FST整个加载到内存。FST只能用于前缀查找。Finite State Transducers（有限状态传感器）。
需要建立索引的时候，将字段的type设置为completion。

高亮

默认高亮使用包裹关键词，可以用pre_tags和post_tags参数替换。

动态词库

ik分词可以设置动态的词库，满足特定的需求。比如根据业务的特性，有些词不需要分词。
在目录/Users/chenpeng/Desktop/elasticsearch-7.7.1/config/analysis-ik下可以看到一份名为IKAnalyzer.cfg.xml的文件。

方便起见，自然是搞个远程扩展字典。重启服务生效，当然对于已经存在的索引，需要重建索引。

重建索引

update by query

现有索引上进行重建。比如把gradeId=1的数据gradeId改成2。

reindex

有时候要改字段格式，或者更换分词插件，这个时候就需要将整个索引重新刷新一遍。es提供了reindex，可以自动重建索引。比如，新建了索引question_v2，将原索引question_v1数据重新导入question_v2。
数据量比较大的话，要等好久。可以异步处理，url后面加上wait_for_completion=false，会返回一个task id。根据这个task id可以随时查看进度。

别名

结合别名，可以方便管理不同版本的索引。比如之前question_v1的别名是question，reindex完毕，将question_v2别名设置为question，再将question_v1的别名去掉。那么所有的question索引CRUD都会转移到question_v2上。

数据建模

在我们的需求中，题目id和知识点id有关联关系。同时为了整个业务的性能，也需要存在知识点的名称。

nested data type

目前知识点是以[{“id”:1,”name”:”有理数”},{“id”:2,”name”:”公式”}]，这种形式存储的。如果我同时搜id=1 and name=公式，是可以搜出来，这个就有问题了。为了解决这个问题，引入了nested数据类型。

查询的时候，需要使用nested关键字，指定path。当然，聚合查询的时候，也一样要加nested关键字，指定path。

父子文档

有种题目叫复合题，比如英语完形填空。大题是一段题干，子题是多个小题。大题和子题结构相似，这种可以搞个父子文档。不过，其实这种情况不是很适合用父子文档结构，博客和评论的存储更合适（评论经常单独更新）。

script

es自带的查询可能不满足我们的需求，于是提供了painless脚本，让我们自己玩。
